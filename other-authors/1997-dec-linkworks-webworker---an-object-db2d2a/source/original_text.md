LinkWorks WebWorker — An Object
Framework that Tames the Web
Introduction
Information Systems (IS) organizations must help their companies become
more extroverted so that MIS efforts directly beneﬁt customers and partners.
Failing to focus on direct beneﬁts to this audience has never been more
dangerous, in this connected world customers can compare competitive
products and services with a click of a button. Companies that have embraced
direct customer interaction ﬁnd extroverted applications consistently provide a
greater return on investment (ROI) than internally focused efforts; not
surprising given the spending MIS has focused on internal operations over the
last two decades. Not only do extroverted projects deliver a high ROI, they also
give your customers and partners more control, and make them an active part of
your organization — in some cases even performing the jobs you once staffed.
To make your IS operations more extroverted requires tighter security, better
control and management, and a ﬂexible system that enables you to be
extremely responsive to your customers’ needs, all attributes that were once
considered mutually exclusive. LinkWorks from Digital Equipment
demonstrates how these apparently conﬂicting requirements can be resolved
using a distributed object framework. Indeed, distributed object frameworks
have now been identiﬁed by Sun, Netscape, IBM, and Microsoft as the
technology that will tame the Internet. These companies have all written white
papers that proclaim the beneﬁts of the distributed object framework, but
LinkWorks can deliver these beneﬁts today. This Proﬁle looks at the beneﬁts
distributed objects offers LinkWorks users, studies how these beneﬁts have
been integrated to the Internet with a new product called WebWorker, and
questions why any company would decide to wait to gain these extraordinary
beneﬁts.
WebWorker – Bringing Distributed Objects to the Web
The Digital LinkWorks environment is already in wide use by many Fortune
1000 ﬁrms to implement document based processes across the enterprise. With
WebWorker, Digital has made it possible for any user with a Web browser to
participate in the LinkWorks environment. To maintain tight security over this
relatively insecure channel, every business process and business object that will
be deployed across the Internet must be explicitly enabled. This approach not
only keeps security tight, it also enables IS to closely manage all aspects of
business processes that are deployed across the Web to customers and partners.
Figure 1 identiﬁes the architecture of WebWorker, which implements the
following functions within the LinkWorks environment:
Integrates Web server security to LinkWorks security;
Establishes LinkWorks’ speciﬁc access control for all Web users;
Programmable access to LinkWorks Business Object Repository using the
Web infrastructure;
Adaptable behavior using Java, ActiveX or scripts; and
Provides an administration screen speciﬁc to Web users and the
WebWorker system.

LinkWorks Turns Desktop Applications Into Mission Critical
Assets
Customers that deploy LinkWorks have discovered that the object-oriented
technology enables them to easily integrate solutions across disparate clients,
servers, and applications, and that application prototyping, deployment,
modiﬁcation, and administration is extremely fast and easy — signiﬁcantly
reducing the total cost of ownership. End-users beneﬁt from LinkWorks
because they can use the desktop applications they have become familiar with
from Microsoft, Lotus, Corel, and others to build new business solutions that
can easily access corporate data, including data resident on legacy systems.
LinkWorks is able to maintain security while using the desktop applications
because data ﬁles, users, and groups of users are all objects in the LinkWorks
system, so access control and data security is tightly controlled by LinkWorks.
Perhaps most importantly, LinkWorks logs the status of every process and
object, even maintaining incremental updates. This capability enables a
company to track the status of every process and to identify what speciﬁc
changes were made to a data ﬁle by whom, and when. In fact, data objects are
easily rolled back to previous versions as required.
When users log-on and are authenticated, they are given access to all
appropriate data objects stored in the infrastructure. Because all objects are
stored centrally, along with the desktop and current status, users can access
their own desktop regardless of where they log in from, or what desktop they
are at. All of these beneﬁts are highlighted in Figure 2. With WebWorker,
LinkWorks can now be extended to Web users anyplace on the Internet,
enabling a company to integrate business partners and customers into the
business process.
The Internet plays catch-up to LinkWorks Technology
It is this underlying object oriented technology that makes LinkWorks so
ﬂexible and powerful, and also positions it to work so extraordinarily well with
Java, JavaBeans, and the object oriented infrastructures being implemented by
Sun, Netscape, and IBM. These vendors are all adopting JavaBeans as local
object components, and have stated they will support CORBA as a mechanism
to let different object oriented environments interoperate with JavaBeans. As
these technologies are further deﬁned and tested, Digital can choose from
several implementations, including:
Adoption of Java as a new object in the current LinkWorks environment;
Create a Java application that delivers the LinkWorks environment to any
platform; and
Integrate Netscape’s future desktop browser (which includes a native
Object Request Broker) as a fully supported LinkWorks environment.
LinkWorks is already tightly integrated to the Microsoft Windows and NT
platforms using the Common Object Model (COM). This Microsoft object
model is referred to as ActiveX in marketing literature, and is evolving to a
distributed model (DCOM). LinkWorks will certainly be able to leverage
DCOM as Microsoft slowly begins to deliver it.
While LinkWorks can integrate to the Microsoft environment today through
COM, the Internet object environment is still too immature to be immediately
leveraged. Indeed Aberdeen research indicates less than 50% of all currently
deployed browsers support Java. As a result, the latest LinkWorks release has
introduced WebWorker, a specialized server that supports and controls access to
LinkWorks from any Web Browser. WebWorker can be used to deploy a users
LinkWorks Desktop directly across the Web, or selectively deploy speciﬁc
components and processes to each Web user.

Digital Equipment Corporation introduced LinkWorks in 1994, and has already
announced its third major release. This puts LinkWorks years ahead in the
deployment of distributed object based solutions, which are highly
differentiated from more traditional Document Management, Workﬂow, and
Groupware products. Digital’s technological lead is possible because
LinkWorks was architected at the outset to utilize distributed object technology,
with components that execute simultaneously on both the client and the server.
By maintaining a consistent distributed object model across desktops and
servers, LinkWorks can track, ﬁnd, and manage any data in its environment,
independent of the application that created it, the platform it resides on, or its
physical location. Users can easily drag and drop objects to share them with
anyone in the workgroup, the enterprise, or even with business partners and
customers via the Internet.
Because LinkWorks integrates to the Windows desktop using OLE automation
which is a component of COM, LinkWorks can easily integrate with all of the
applications users have already installed. By leveraging desktop applications as
the user interface to enterprise business solutions, LinkWorks is able to
signiﬁcantly lower end-user and administrative staff’s training. And by
incorporating users, and groups of users, as objects within the framework,
LinkWorks extends its tight access control and security model to the desktop
applications. Users who log-on have controlled access to all data objects stored
in the LinkWorks infrastructure, and because the objects are stored centrally,
users can access their own desktop regardless of where they are or what system
they log in from. Since the data for LinkWorks processes are maintained
centrally, access is tightly controlled, the status of the data object is always
known, and the status of the work process is deﬁnitive. With WebWorker, any
Web-enabled user can participate in the LinkWorks environment, enabling
LinkWorks customers to integrate their business partners and customers into
the company’s business process.
The Development Environment
LinkWorks developers build applications in this object oriented environment
with "components." Even sophisticated solutions are created from a toolkit
which provides many of the most frequently used classes and components. This
environment makes development easy, because a new function can often be
developed by simply changing the conﬁguration or modifying a current
component. For particularly difﬁcult projects however, new components may
be built from scratch, representing a new class of object. An important feature
of the LinkWorks environment is that once a component is deﬁned, it can be
used in any solution that needs it. If that component is modiﬁed at a later date,
all solutions that use the component are "updated" automatically. Perhaps one
of the most important features of this environment is that current applications
can be encapsulated as objects, and then used like any other objects to create
new solutions.
Establishing the Framework
LinkWorks software executes simultaneously on the desktop and server. Today,
the LinkWorks server is supported on Digital UNIX , HP-UX, OpenVMS, IBM
AIX, and Windows NT (3.51, 4.0, and Alpha). The desktops currently
supported include MS Windows 95, MS Windows NT client, MS Windows,
Motif (Digital, Sun, HP, and IBM), the Macintosh OS (68000 and PowerPC),
and WebWorker enables all the popular Web browsers to participate as well.
LinkWorks clients, and WebWorker server, use RPCs to communicate with
LinkWorks servers for added reliability, recoverability, and manageability. The
client manages the user interface and maintains a consistent look and feel
across all the clients supported. In addition, LinkWorks can record or execute
LinkWorks scripts, making it very easy for users to automate tasks. Indeed, a
LinkWorks script can be executed on any desktop, or on any server. If desired,
it can also be encapsulated and used as an object in any LinkWorks solution
from any desktop. The desktop also integrates LinkWorks with the local
operating system and ﬁle sub- system, and makes LinkWorks objects available
to applications that run on the desktop. For Windows desktops, there is also a
very impressive use of OLE linking that allows Windows applications to
become objects in the LinkWorks repository (See Windows Integration).
On the server, LinkWorks tracks each desktop based on speciﬁc user ID and
can also establish basic desktop templates based on a group ID. The full object
oriented system executes on the server and is used to create, execute, distribute,
and manage all LinkWorks objects and their constituent components
(applications, data, attributes, and methods). Every object is given an object
identiﬁer which is globally unique, and every instance of an object is saved
along with the data that speciﬁes who modiﬁed it, when it was modiﬁed, and
what the modiﬁcations were. This powerful function allows a ﬁle to be made
available for group modiﬁcation, and maintains a complete log of who made
what changes when, permitting any intermediate revision to be captured. The
server also maintains a complete record of every object, no matter what server
the object is located on. This allows searches to be made that can ﬁnd any
object in the LinkWorks environment, regardless of its location.
The server software integrates to the users database of choice, which is used as
the systems repository. Currently LinkWorks supports Oracle, Microsoft SQL
Server, CA-Ingres, or Informix as repository databases, and offers several
sophisticated tools to manage system resources. Of particular interest to

Microsoft centric organizations is that Digital is currently working to become
BackOfﬁce Certiﬁed. This certiﬁcation will guarantee customers that
LinkWorks installs and integrates seamlessly into the full Microsoft
environment, including utilities that provide performance monitoring, system
conﬁguration, and user directories, as well as directly using other BackOfﬁce
applications such as SQL Server.
Department and End-User Beneﬁts
Out of the box, LinkWorks adds information sharing, electronic mail, document
management and workﬂow capabilities to existing systems and applications.
Current end-user applications can be easily integrated into the LinkWorks
framework to become part of the overall groupware environment.
However, the very ﬁrst beneﬁt end users will likely appreciate is that
LinkWorks remembers their desktop conﬁguration. When end users log in, the
LinkWorks environment remembers the previous state of the desktop, and re-
creates that desktop regardless of where the user logs in from, or what desktop
the user is on (Windows, NT, Motif, etc.). A traveling user that leaves a
Windows desktop in the United States, and logs in from a Browser in Belgium,
will see the desktop just as it was left, with all of the same data and ﬁles,
assuming the administrator set the permissions to allow this. If the desktop has
objects on it not native to the environment (a Word object on a browser for
example), the object can be downloaded to the browsers underlying ﬁle system.
Once in the LinkWorks environment, desktop users have an option, they can
continue to use their desktop applications as they always did, or as a
LinkWorks participant, they can begin using the collaborative tools available to
them. For example, a word processing document can be made available to
anyone in the LinkWorks environment for editing. LinkWorks makes sure
everyone gets his or her own copy of the data, and synchronizes all edits. At
any moment, the exact status of the document can be ascertained, and the
originator of any edits identiﬁed.
If the user wants to take a more proactive approach, LinkWorks allows the user
to specify a route. Using a graphical interface, the user selects the ﬂow of the
object from an organization chart. Each recipient is dragged and dropped into
the path. If desired, branches can be assigned based on any number of
conditions. Simple graphical objects allow all of this to be accomplished in an
amazingly easy and non-threatening fashion. For other users who might want to
stay in touch with what’s happening on speciﬁc projects, the environment
allows a user to identify objects in which they are interested. If that object is
modiﬁed, the user is notiﬁed and given the opportunity to view the updated
object. LinkWorks does this for ANY object — an Excel spreadsheet, an Access
database, or any other object encapsulated within LinkWorks.
MS Windows Integration
Since Windows is the predominant desktop in most companies, Digital has
established a tight integration with the Windows environment. Digital
Equipment and Microsoft have worked together on several joint efforts,
including the porting of NT to the Alpha platform and the integration of OLE
and distributed object technology with Digital’s object broker product.
LinkWorks clearly demonstrates the power of linking distributed objects and
COM technologies. For example, LinkWorks can use OLE Automation to
invoke actions on the desktop, allowing LinkWorks to tightly integrate with
OLE capable applications that operate in the Windows environment. Another
powerful example of what can be achieved by this tight integration is currently
demonstrable when a user drags a LinkWorks folder onto their Windows 95
desktop. While the folder looks like any other Windows folder, it behaves as a
LinkWorks object. For example, the simple act of opening the folder is a
LinkWorks action, which can trigger a LinkWorks business process action.
Windows 95 folders have no such support, but by closely integrating the
LinkWorks object model to COM, LinkWorks capabilities can be brought to the
Windows desktop.
MIS and Support Beneﬁts
The LinkWorks distributed object architecture enables rapid application
development and deployment. Perhaps more importantly, it also enables
changes to be deployed instantly, and makes the incremental modiﬁcations
typically needed to meet local department needs extremely easy; easy enough
that it can often be done by the department through simple conﬁguration
changes.
Another beneﬁt of the LinkWorks environment is that data and applications
only need to be integrated into LinkWorks once in order to be available to
anyone in the corporation. The LinkWorks toolset includes templates for many
databases and desktop applications, so this encapsulation will move quickly.
However, MIS still needs to do its homework and manage this complex effort.
Moreover, large companies typically have several legacy applications and
databases that were "frozen" years ago. While the LinkWorks Application Plus
Objects (APO) interface makes this possible, it does require a developer who
understands both objects and the legacy application.

Aberdeen recommends companies confront these difﬁculties for two reasons:
ﬁrst, because this effort is extremely valuable. Once properly integrated into
LinkWorks, the legacy application can be tied into any number of LinkWorks
solutions, further leveraging the company’s investment in it. Secondly, legacy
programmers are sure to appreciate the opportunity to become experts on object
oriented environments like LinkWorks. While their ramp-up time may be slow,
once up-to-speed, they can be re-assigned to creating more mission critical
applications in LinkWorks, creating tremendous forward momentum for re-
engineering efforts.
Once these difﬁculties are resolved and the legacy environments are integrated
into LinkWorks, several other seemingly intractable problems get resolved. For
example, LinkWorks will:
Fully implement security across a workgroup environment;
Integrate with existing mail clients to produce a robust messaging system;
Provide an object repository of all data and resources for easy backup and
recovery;
Enable complex searches against all the business objects in the network;
Leverage end-users’ existing desktop and application expertise;
Leverage current training by allowing users and MIS to stay in their
favorite environments;
Maintain objects across several databases and applications;
Provide ability to centrally manage most performance characteristics; and
Make Microsoft OLE Automation work in a distributed environment.
Managing Distributed Objects — Experience Counts!
While all the major vendors are now scrambling to implement products
architected to use distributed objects, Digital has been delivering such a product
for over three years, and have gained signiﬁcant real-world experience from
having done so. For example, Digital has found that customers save incredible
time and money when using the distributed object framework to prototype,
deploy, maintain, manage, and administer solutions. Interestingly, Digital has
also found that while prototyping is simple, deploying that prototype demands
another whole level of effort. This is the result of the prototype operating as a
single instance, while the solution that runs in the enterprise must be executed
as multiple instances, which introduces a whole new level of complexity. This
complexity can be eased somewhat if the vendor has the appropriate utilities
and services, and Digital has been developing and deploying these for the last
three years, making it unlikely similar features will appear in competitive
systems anytime soon.
As an example, LinkWorks incorporates a Transaction Dispatch Facility which
helps IS optimize the use of server processes and existing network resources.
The facility enables a pool of server tasks to be established on each server and
dynamically adjusted to meet the workload. As a result, IS is able to dispatch
tasks to speciﬁc servers, and assign priorities to each task so that the
performance for end users can be tuned. This service enables new Domains to
be added to the environment and tasks easily distributed to them. Another
common problem encountered in real-life enterprise scenarios is the need to
support multiple object stores in a seamless fashion across the enterprise.
LinkWorks has addressed this through the Administration Manager which lets
the entire LinkWorks environment be controlled on a per Domain basis from a
central location. The Domain architecture is important because it enables
application availability while also controlling both network utilization and
object performance. Application availability is possible because the Domain
Cell holds the LinkWorks objects, and multiple Domain Cells are combined to
form a cluster on speciﬁc servers. IS can then select which servers will support
which Domains. By doing this, any server failure will only affect the Domain
Cells it runs, so the clients attached to that server still function, although they
will not be able to access the objects stored there. As a result, a server failure
only has a local impact, and does not bring down the entire LinkWorks
Domain.
To provide robust security, every LinkWorks object is assigned a globally
unique identiﬁer and speciﬁc access rights when it is created that deﬁne what
actions can be performed on that object, and by whom. Administrators deﬁne
access rights for groups of users, individual users, or a combination of the two.
LinkWorks supports a highly granular approach to access control, allowing a
systems administrator to create custom access rights as well as using up to eight
pre-deﬁned access controls. The globally unique identiﬁer guarantees objects
remain unique, even if deployed across multiple LinkWorks systems.
Enterprise customers have also found that deploying object oriented solutions
to different systems is extremely complex because every implementation may
have locally unique features. LinkWorks has solved this problem with a truly
remarkable feature. LinkWorks solutions can be "grouped" or "packaged" into
software components, and installed or de-installed across any number of

LinkWorks servers or entirely separate and disconnected systems. Since every
object is globally unique, inadvertent replication of objects is impossible, and
objects are easily referenced across the entire enterprise or even public domain.
In addition, Digital has made it possible for these solutions to be distributed
through any media, including disk, tape, CD-ROM, or even the corporate
messaging infrastructure.
Conclusion
Every enterprise today is under pressure to reduce costs and better serve their
constituents. Aberdeen research indicates the fastest and least expensive
method to accomplish this is to bring those users into the business processes of
the enterprise itself. Not only does this offer a high return on investment, it also
tightly integrates the users to the infrastructure which enables the enterprise to
better track, manage, and serve the users expectations. This tight integration
also enables the enterprise to collect valuable insights into the users behavior
which is extremely valuable information for sales, marketing, and production
forecasting.
LinkWorks is an enterprise ready solution that uses leading edge technology to
enable rapid deployment of these sophisticated business processes wherever
they need to go, without sacriﬁcing control or security. The technology used by
LinkWorks is so advanced that Digital has already gained three years of
experience deploying a distributed objects framework while Netscape, Sun, and
IBM promise to release products sometime in the next 6 — 12 months.
Aberdeen urges companies to evaluate LinkWorks carefully. At a minimum, IS
can learn why every vendor is rushing headlong to support distributed objects,
but more likely, IS will recognize the beneﬁts as critical to corporate success,
and will begin implementing solutions using LinkWorks today.

### Abstract

Aberdeen Group profile of Digital Equipment Corporation's LinkWorks WebWorker — a distributed object framework enabling enterprise document management and workflow to extend to the Web via any browser. The study highlights DEC's three-year head start in distributed object deployment over Sun, Netscape, IBM and Microsoft who had only white papers. Evaluates architecture, security, ROI, and recommends enterprise IS organizations evaluate LinkWorks for extending business processes to customers and partners.

### Prescience Detail

**Correct Predictions:**
- Java and JavaBeans did become dominant enterprise programming technologies as predicted
- Microsoft's COM/DCOM/ActiveX remained important for about a decade before .NET superseded them
- Extroverted applications (customer/partner facing) delivering higher ROI than internal systems proved accurate
- The concept that distributed object frameworks would tame enterprise web integration was directionally correct

**Incorrect or Partially Correct Predictions:**
- CORBA did not become the dominant interoperability standard; web services (SOAP) and later REST APIs replaced distributed objects
- Netscape's native ORB-equipped browser never shipped as predicted; Netscape was acquired by AOL in 1999 and discontinued in 2008
- DEC's three-year head start did not translate to market leadership; DEC was acquired by Compaq in 1998 and LinkWorks was eventually discontinued
- The specific 6-12 month timeline for Netscape ORB was not achieved

**Long-Term Outcomes:**
- DEC was acquired by Compaq for $9.6B in June 1998; Compaq acquired by HP in 2002
- LinkWorks was discontinued; SharePoint (Microsoft, 2001) became the dominant enterprise collaboration/document management platform
- CORBA was widely adopted in the late 1990s but superseded by SOAP web services by 2003-2004 and REST by 2007
- Digital UNIX renamed Tru64 UNIX; HP ended support in 2012
- OpenVMS survived and is maintained by VMS Software Inc. (VSI) on x86 as of 2026
